# Greeting with Remote Import (DAGL Workflow)

Demonstrates importing an **externally deployed Lambda function** into a unum
workflow using the `@import` directive. The original remote function is never
modified — the compiler generates a proxy Lambda that calls it.

## Architecture

```
[Standalone Greeter Lambda]         [Unum Workflow SAM Stack]
  (deployed separately,              Greeter(proxy) → Formatter
   its own stack)                        |
                                         ↓ boto3.invoke()
                                   [Original Greeter Lambda]
```

## Source files (what you write)

```
greeting-with-remote-dagl/
├── workflow.dag                    # DAGL workflow with @import directive
├── Formatter/app.py               # Local function with @input/@output annotations
├── input.json                     # Sample input for testing
└── standalone-greeter/            # Individual function (separate project)
    ├── app.py                     # Greeter function code
    └── template.yaml              # Its own SAM template for independent deploy
```

Note: there is NO `Greeter/` directory in source — it is generated by the compiler.

## Step-by-step

### 1. Deploy the standalone Greeter function

This is an individual Lambda deployed independently (its own SAM stack):

```bash
cd standalone-greeter
sam build
sam deploy --guided --stack-name standalone-greeter
```

After deployment, copy the ARN from the output:

```
Key: GreeterFunctionArn
Value: arn:aws:lambda:eu-central-1:683003725669:function:standalone-greeter-GreeterFunction-XXXX
```

### 2. Update workflow.dag with the real ARN

Edit `workflow.dag` and replace the placeholder ARN with the real one from step 1:

```
@import("Greeter", "arn:aws:lambda:eu-central-1:683003725669:function:standalone-greeter-GreeterFunction-XXXX")
```

### 3. Compile the workflow

```bash
cd ..
unum-cli compile --platform dagl --workflow workflow.dag
```

This generates:

- `Greeter/app.py` — **proxy** that calls the remote Greeter via boto3 (generated!)
- `Greeter/unum_config.json` — orchestration config for the proxy (generated!)
- `Formatter/unum_config.json` — config for the local function (generated!)
- `unum-template.yaml` — SAM template with both functions (generated!)

**Generated files after compile:**

```
├── Greeter/                       (entire directory generated)
│   ├── app.py                     (proxy → remote ARN)
│   └── unum_config.json
├── Formatter/unum_config.json     (generated)
└── unum-template.yaml             (generated)
```

### 4. Generate SAM template + Build

```bash
unum-cli template
unum-cli build
```

Or in one command:

```bash
unum-cli build -g
```

### 5. Deploy the workflow

```bash
unum-cli deploy
```

This deploys both the proxy Lambda (Greeter) and local Lambda (Formatter)
as a single SAM stack. The proxy gets `AWSLambdaRole` policy so it can invoke
the remote Greeter function.

### 6. Invoke (optional)

```bash
unum-cli invoke -f input.json --wait
```

## Type checking with Redis (optional)

If the standalone Greeter was deployed with `--redis-url`, its type signature
is stored in Redis. Use `--registry` during compile to validate types across
the remote boundary:

```bash
unum-cli compile --platform dagl --workflow workflow.dag --registry redis://localhost:6379
```

or if connect it woth api gateway then invoke:

```bash
Invoke-RestMethod -Uri "https://7ef8lb2x43.execute-api.eu-central-1.amazonaws.com/greet" -Method POST -Body '{"name": "Alice", "style": "enthusiastic"}' -ContentType "application/json"
```
